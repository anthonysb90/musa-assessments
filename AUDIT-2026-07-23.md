## 1. Executive Summary

**The short version:** You built something real. The product voice is consistent, the pastoral instincts (withhold-and-reveal, care alerts, anonymity minimums) are genuinely differentiating, and two of your reports are close to premium quality. But the platform has **three launch-blocking security holes verified in the live database**, the reports portfolio is wildly uneven (2 strong, 8 thin), and about half your emails look like debug output.

**The good news:** The live database has **0 sessions, 0 results, 0 access grants**. Nothing has been breached and no customer has been shortchanged. You are pre-launch. Every critical item below can be fixed before it ever matters.

| Area | Grade | One-line verdict |
|---|---|---|
| Security | **F (fixable to A)** | Payment bypass + full PII read are live right now. Both are half-day fixes. |
| Database schema | **B** | Base schema is better than the repo suggests (good constraints/indexes exist). Migration hygiene and a few races need work. |
| Emails | **C-** | One good branded template exists but only half the emails use it. Several must-have emails don't exist. |
| Reports | **C+ overall, B+ ceiling** | Kingdom Design and Leadership Style prove you can hit $100 quality. The other eleven are not there. |
| App UX | **B-** | Warm and coherent, but no answer autosave is a critical gap, and there is one real money-display bug in checkout. |
| Admin | **B** | Strong analytics for a v1. Missing the operational tools (resend, refund, audit log) that will eat your support time. |

**Answer to your main question ("is this a $100 report or a Cosmopolitan quiz?"):** It is neither. It is honest, scripture-grounded, and better designed than any church assessment tool I would compare it to. But it is not yet worth $100, for one specific reason: **almost all content is band-lookup, not combination-aware.** A $100 report (CliftonStrengths 34, Birkman, Working Genius) generates text conditioned on your unique *combination* of scores. Yours mostly serves the same paragraph to everyone in the same band. Section 5 spells out exactly how to close that gap. It is a content-writing problem more than a code problem, and it is very doable.

## 2. Launch Blockers: Security (fix before promoting the site)

These were all **VERIFIED LIVE** against production unless noted.

### S1. CRITICAL: Anyone can mint paid access for free (payment bypass)

**Verified live:** `finalize_purchase()` and `redeem_free_order()` are SECURITY DEFINER functions, executable by the `anon` role, and **neither contains any authorization check** (I checked the live function bodies). The Stripe verification lives only in the Next.js route. The RPC itself trusts whatever it is handed.

Anyone who opens your site's JS bundle gets the public anon key, then calls:

```js
supabase.rpc('finalize_purchase', { p_pi: 'fake_' + Math.random(), p_email: 'x@x.com',
  p_name: 'x', p_slugs: ['pastor-profile'], p_seats: 100000, p_kind: 'assessment',
  p_bundle: null, p_church: null })
```

That inserts a real `access_grants` row with a working code and 100,000 seats. Every paid assessment becomes free. `p_seats` is floored at 1 but never capped.

**Fix:** Revoke anon/authenticated EXECUTE on `finalize_purchase`, `redeem_free_order`, `redeem_coupon`, and `consume_seat`. Call them only from server routes using the **service-role key** (create a `supabaseAdmin` client in `app/lib/supabaseServer.js` using `SUPABASE_SERVICE_ROLE_KEY`). Also cap `p_seats` inside the function. Full work order: **WO-1**.

### S2. CRITICAL: Every anonymous taker's results and PII are publicly readable

**Verified live**, these two policies exist in production:

```sql
-- sessions: "anon sessions readable"  USING (profile_id IS NULL)
-- results:  "anon results readable for tokenized sessions"
--           USING (EXISTS (... s.id = results.session_id AND s.profile_id IS NULL))
```

Neither policy references `result_token`. The token is only a WHERE filter in your own app's query. Anyone with the anon key can paginate the entire `sessions` table (getting every `result_token`) and the entire `results` table. `results.scored_json` embeds the taker's name, email, phone, age band, and full psychological scores. Since standard assessments require no login, that is nearly every taker.

This also silently breaks the church "withhold results, reveal in person" promise (see S5).

**Fix:** Drop both policies. Replace with a SECURITY DEFINER RPC `result_by_token(p_token)` that returns exactly one result for an exact token match, and convert the results page to fetch through it. Full work order: **WO-2**.

### S3. HIGH: Couples, circles, and team tables are fully public

**Verified live:** `couples`, `review_circles`, and `rater_groups` all have `FOR SELECT USING (true)`. Any anon client can dump every row: spouse names, subject emails, full `self_json` and `spouse1_json/spouse2_json` scored assessments, and all the invite codes. The Supabase advisor also flags the `WITH CHECK (true)` inserts on `couples` and `rater_groups` (unlimited anonymous row creation).

**Fix:** Drop the three `using (true)` SELECT policies. Serve these flows through code-keyed SECURITY DEFINER functions that return only what the flow needs (aggregate for teams, both-done payload for couples). Move creation into RPCs so you can rate-limit. Full work order: **WO-3**.

### S4. HIGH: `/api/invite/send` is an open, unauthenticated email/SMS relay

No auth, no Turnstile, no rate limit. It accepts `{ to, name, link, context, fromName }` from anyone and sends a Mission USA-branded email (from your authenticated `assessments@chchurch.com` domain) or a Clearstream SMS to any address, with an **arbitrary attacker-supplied link as the CTA button** and unescaped HTML in `name/context/fromName`. This is a turnkey phishing kit that will also burn your sending domain's reputation and your Clearstream credits.

Related: `/api/auth/magiclink` is also unauthenticated and uses the service-role key to create confirmed users and email sign-in links for any address anyone submits. That is an email-bombing and account-pre-provisioning vector.

**Fix:** Accept a code (`team_code`/`couple_code`/`circle_code`), look it up, and build the link server-side. Never accept a raw URL. Escape all user strings. Add Turnstile plus per-IP rate limiting to both routes. Full work order: **WO-4**.

### S5. HIGH: "Withhold results / reveal in person" is client-side only

The results page fetches the full `scored_json` to the browser first, then React decides whether to show the "your pastor will review this with you" screen. The data is already in the network response, and with S2 the taker can just query the table directly. The withhold promise you sell to church partners is not actually enforced anywhere.

**Fix:** Enforce withholding inside the `result_by_token` RPC from WO-2. If the session's church assignment has withhold enabled and the caller is not an admin or that church's leader, return the blocked state with no scores. This is bundled into **WO-2**.

### S6. MEDIUM: Open redirect in the auth callback

`/auth/callback?next=https://evil.com` redirects there after sign-in, because `new URL(next, origin)` honors absolute URLs. Combined with S4's email tricks, it makes convincing phishing links on your real domain.

**Fix:** Accept only relative paths: `next.startsWith("/") && !next.startsWith("//")`, else fall back to `/welcome`. Same validation in the magiclink route. Part of **WO-4**.

### S7. MEDIUM: Turnstile gaps and no rate limiting anywhere

`verifyTurnstile` returns `ok: true` when unconfigured **and on any network error** (fails open). The `team_code` and `couple_code` branches in `/api/submit` return before the Turnstile check ever runs, and `/api/circle/observe` has no check at all. There is no rate limiting on any POST route. That means group aggregates can be ballot-stuffed by a script, which corrupts the reports pastors will act on.

**Fix:** Move the Turnstile check to the top of `/api/submit`, add it to `circle/observe`, and add Upstash/Vercel KV rate limiting to all POST routes. Part of **WO-4**.

### S8. MEDIUM: Client-supplied `church_id` on submit

`/api/submit` writes `profile.church_id` verbatim and uses it to email that church's leaders. Anyone can attribute fake completions to any church and spam its recipients. **Fix:** resolve the church from the validated `assignment_token` only. Part of **WO-5**.

### What checked out fine (verified live, so you don't chase ghosts)

- Checkout pricing is computed **server-side** from the DB. The client cannot set the Stripe amount. Coupon validation and platform fee math happen on the server. This is the correct pattern.
- Admin-managed tables (`assessments`, `items`, `churches`, `church_assessments`, `church_users`, `admin_users`) all have live RLS requiring `is_admin()` for writes. A non-admin cannot tamper with prices via the anon key.
- `access_grants`, `circle_responses`, and `coupons` have RLS enabled with **no** client policies (server-function access only). That is the right model; the advisor INFO flags on these are intentional design, not bugs.
- `results.session_id` is UNIQUE live, `sessions.result_token` is UNIQUE and indexed, `responses(session_id, item_id)` is UNIQUE, `church_assessments(church_id, assessment_id)` is UNIQUE, `church_users(church_id, email)` is UNIQUE. The base schema is more sound than the repo's partial migration folder implies.

## 3. Database

### What is sound

Postgres 17, healthy project. Consistent `timestamptz`, idempotent migration style, quarantined `wellbeing_results` with tight policies, singleton `platform_settings` with a bounded fee, unique constraints in the right places (see list above). The intended security model (SECURITY DEFINER functions + no-select tables) is correct where it was fully applied.

### Issues to fix

1. **Source of truth is the live DB, not the repo.** The repo has migrations 09 through 31 only (no 01-08, no 12), and the coupon function bodies were applied ad hoc and exist only as comments. You cannot rebuild this database from the repo, and you cannot code-review what is not in it. **Fix in WO-6:** dump the live schema into the repo and adopt tracked migrations.
2. **Seat and coupon races.** `check_access` (at start) and `consume_seat` (at completion) are decoupled, so N people can pass the check and all complete on the last seat. `consume_seat` failures are swallowed and there is no idempotency, so a double-submit burns two seats. Coupon validate and redeem happen at different times (classic check-then-act window). **Fix in WO-6.**
3. **Missing indexes (verified live):** `circle_responses.circle_code` has no index (used by every circle aggregate), `sessions.cohort` has no index (used by team/couple recompute), `access_grants(lower(email))` has no index. Small now, painful later.
4. **String joins with no FK:** `circle_responses.circle_code`, `sessions.cohort` join to codes by raw text. Typos insert silently; deletes orphan rows.
5. **Email casing:** `access_grants.email` is stored as-is while policies compare `lower()`. Move email columns to `citext` or normalize on write.
6. **No Stripe webhook.** Donations are never recorded anywhere, and there is no server-side ledger of payments. Add a `payment_intent.succeeded` webhook with signature verification. **WO-7.**
7. **Unused `resume_token` column on `sessions`.** The schema anticipated resume; the app never implemented it. WO-8 puts it to work.
8. **No audit trail** for admin actions, seat consumption, or coupon redemptions. For a system handling money and pastoral data, add an append-only `audit_log`. **WO-12.**

## 4. Emails

### Inventory of what sends today

| Email | Trigger | Template quality |
|---|---|---|
| Magic-link sign-in | /api/auth/magiclink | Branded shell |
| Results ready (3 variants: full, sensitive, link-only) | /api/submit | Branded shell, includes inline score summary (nice) |
| Purchase / access code | /api/checkout/finalize | Branded shell |
| Participation invite (team/couple/circle) + SMS | /api/invite/send | Branded but the shell is **copy-pasted inline** and already drifting |
| Assessor notification | /api/circle/create | Copy-pasted shell |
| Church admin invites (two near-identical routes) | invite routes | **Bare unstyled `<p>` tags** |
| Church completion notice | /api/submit | Bare `<p>` tags |
| Care alerts (couple safety, pastor wellbeing) + partnership request | /api/submit, /api/partner-request | Bare `<p>` tags |

The problem is not that you lack a good template. `shell()` in `email.js` is a legitimate 600px branded transactional skeleton. The problem is only 5 of ~12 email bodies use it. A church leader's **first** email from your platform (the dashboard invite) is the ugliest thing the system sends.

### Emails that should exist and do not

- **Payment receipt** (amount, date, last4). The access email is not a receipt, and churches expense these purchases. Cheapest fix: set `receipt_email` on the PaymentIntent and let Stripe send it.
- **Couple both-done notice.** `record_couple_member` already returns `both_done` and nobody gets emailed. Spouses must remember to come back. Same for circle-complete and team-threshold-reached.
- **Welcome email** on first sign-in.
- **Weekly church digest.** Today the church contact gets one email per member completion. A membership class of 40 means 40 emails in an afternoon. They will filter you to spam. Make per-completion opt-in and send a weekly summary instead.
- **Unused-seats nudge** to multi-seat purchasers.

### Quality and deliverability problems

- **No HTML escaping anywhere.** Every template interpolates user-supplied names raw. Combined with S4 this is exploitable; even alone it lets content spoofing into staff inboxes.
- **Send failures are silently swallowed.** Most routes ignore the `sendEmail` result inside try/catch with no logging. A wrong API key means every email silently vanishes, **including the couple-safety and pastor-wellbeing care alerts.** For care alerts, silent failure is a human-safety issue, not a UX issue.
- **Checkout re-finalize duplicates the access email** on every refresh of the return page.
- No preheader text, no `<head>`/dark-mode handling, no MSO conditionals, no physical address or unsubscribe affordance (the "what to take next" promo block makes the results email arguably promotional under Gmail/Yahoo bulk-sender rules).
- `APP_URL` falls back to `musa-assessments.vercel.app`. If the env var is ever unset, every link and logo in every email points at vercel.app while sending from chchurch.com. Classic spam-filter trigger. Verify SPF/DKIM/DMARC alignment for the sending domain in Emailit while you are at it.

All of this is one work order: **WO-9** (single `renderEmail()` wrapper + `esc()` + logging + the missing emails).

## 5. Reports: The $100 Question

### Honest verdict by assessment

| Tier | Assessments | Current value | Why |
|---|---|---|---|
| **Near-premium** | Kingdom Design, Leadership Style | $49-79 | Kingdom Design: 1,200+ words per type, biblical mirror stories, written prayers, 30-day checklist. Leadership Style: the best *architecture* in the app: answer playback, slot-filled 90-day plan, pairing table, coaching guide. |
| **Solid mid** | Big Five, Spiritual Gifts, Forgiveness | $25-39 | Real content per band, honest scoring language, but no interactions and gifts content is identical for everyone (only rank changes). |
| **Thin** | DISC, Fivefold, Enneagram, Rooted, Leadership Health, Spiritual Growth, Church Growth, Planter | $0-15 | DISC total shown content is roughly one screen. Rooted-family "next steps" are one identical sentence whether you scored 1.2 or 3.9. Church Growth's whole result is a single paragraph. |

### What is already genuinely good (keep and build on)

- Real typographic identity: Fraunces + Inter, **self-hosted as static weights specifically so PDFs embed real outline fonts**. That is print-designer thinking most SaaS never does.
- Bespoke score-driven visuals: the Leadership stool SVG that fills to your scores, the Big Five radar with labeled vertices, the illuminated Spiritual Gifts triptych.
- A serious print stylesheet: letter-size @page, widows/orphans control, running footer, gift studies force-expanded in print.
- Honest scoring. No fake percentiles anywhere. Reverse-scored items handled. The Church Planter instrument has real attention checks and a social-desirability count shown only to the assessor, which is exactly right.
- Scripture integration that no competitor has: anchor verses, devotions, written prayers, tone discipline ("developmental, never a verdict").

### Why it is not $100 yet (the five specific gaps)

1. **No interaction content anywhere.** Zero paragraphs conditioned on combinations: no "high Openness + low Conscientiousness," no gift-constellation themes, no Enneagram wings or arrows, no DISC intensity narrative. This is the single dividing line between a personalized instrument and a band-lookup quiz. It is a writing project, not a code project.
2. **Your own design system is dead code.** `reportTokens.js` is an excellent locked token file (type scale, spacing grid, color-blind-safe score shapes, a written chart contract). **The report page never imports it.** Instead: 187 inline fontSize declarations across ~26 distinct sizes, hardcoded hexes everywhere, no tabular numerals on score columns, bars with no tick marks or band-cutoff reference lines, and "Needs Attention" rendered in the *palest* color on the page (it disappears on greyscale office printers).
3. **No document architecture.** One long scroll of cards. No cover page, no table of contents, no page numbers, no section openers, no closing summary. The code itself admits page numbers "require the headless print route," which does not exist. `break-inside: avoid` on whole sections punts entire sections to the next page, leaving big white voids: the classic HTML-to-PDF tell.
4. **Print bugs that lose content.** Enneagram type profiles and Forgiveness subscale details render only when clicked open, and unlike gift studies they have **no print expansion rule**. The printed Enneagram report omits all nine type profiles unless the user happened to expand them. The 13-spoke Planter radar has **no spoke labels**, so a reader cannot tell which vertex is which. Report hero images load from a third-party CDN and Pexels at runtime.
5. **Uncalibrated cutoffs.** Everything is raw-sum percent with author-judgment bands (High = 70+). Church samples are agree-heavy; most takers will pile into "High/Strength" and the report will feel flattering-generic. You are already storing every response. After ~500 takers per instrument you can replace fixed cutoffs with empirical tertiles and honestly print "compared with N ministry leaders." That one line is the cheapest credibility upgrade available to you.

Also worth flagging: silent tie-breaks manufacture false certainty. DISC ties resolve by a fixed D>I>S>C priority, Kingdom Design ties default to the A pole, and the maturity-level matrix deliberately breaks ties toward the more flattering level. A tied result should print a "close call" note (you already do this for Enneagram; extend the pattern).

### The path to $100 (in order)

1. **WO-10a:** Write interaction content. Big Five: 10 trait pairs x 4 quadrants = 40 paragraphs. Gifts: 15-20 top-3 constellation themes. Enneagram: wings + growth/stress arrows (~18 blurbs). DISC: per-dimension narrative. This is the money.
2. **WO-10b:** Clone the Leadership Style architecture (answer playback, 90-day slot-filled plan, pairing table, coaching guide) into every paid report. The pattern exists; it is a copy job.
3. **WO-10c:** Wire `reportTokens.js` into the renderer, fix the print bugs, label the Planter chart, localize images.
4. **WO-10d:** Build the server-side PDF route (Playwright/Chromium): cover page, TOC, numbered pages, emailed attachment. This is what makes it *feel* like a $100 artifact.
5. **WO-10e:** Cross-assessment synthesis page ("Your Kingdom Design + your gifts + your leadership style point at..."). The data is already joined by email. **No competitor has this.** For a scripture-centered platform it is the signature feature.

## 6. App UX

### Critical

1. **No answer autosave.** Answers live only in React state. A refresh, back-swipe, or iOS tab eviction at minute 18 of a 20-minute assessment destroys everything. Worse, **a failed submit also loses everything**: the error state offers no retry, only "Back to assessments." And the dashboard shows a "Resume" link that actually restarts from question 1. The DB already has a `resume_token` column waiting to be used. **WO-8.**
2. **Checkout shows the wrong price in the pay phase.** `dollars()` divides by 100, and the pay phase calls `dollars(bd.subtotal / 100)`: a double division. The pay screen and the "Pay $X" button can show 1/100th of the real amount ($0.29 instead of $29.00). Charging is server-driven so nobody is mischarged, but a payment button with the wrong number destroys trust. Also: the continue button does not disable while the intent is being created (double-click risk). **WO-11.**

### High

3. **Details gate friction.** After finishing all questions, anonymous takers hit a required first/last/email/**phone**/age form with a silently disabled button and no inline validation messages. Required phone at the moment of maximum anticipation is a known drop-off driver. Make phone optional and say what is missing. Paid takers also re-enter everything they already gave checkout.
4. **Mobile navigation is nearly empty.** The mega menu, "Home," and "For churches" links are all `display:none` under 820px with no hamburger replacement. Most church traffic is mobile.
5. **Accessibility:** answer buttons are bare `<button>`s with no radiogroup semantics, no `aria-checked`, selection conveyed by color only, no visible focus styles anywhere (a side effect of the inline-style approach). A screen reader user cannot take these assessments.
6. **Lost report links are unrecoverable.** Most takers are anonymous; if they lose the email there is no self-serve or admin resend. This will be your #1 support request. **WO-12.**

### Medium

7. Homepage is fully client-rendered; the most-shared URL on the platform serves "Loading assessments..." to crawlers and link previews (I confirmed this on the live site). Make it a server component with client islands.
8. 25 questions per page is heavy on mobile; default to 10.
9. Every loading state platform-wide is a bare "Loading..." paragraph.
10. `results/[token]/page.js` is 2,330 lines and ships every report renderer plus 200KB+ of content libraries to every taker. Split per-report components with dynamic imports.

## 7. Admin: What You Have and What Is Missing

### What you have (this part is a strong v1)

KPIs with date ranges, weekly chart, funnel, breakdowns by assessment/church/role/age/source, top gifts, planter readiness, a pastor wellbeing care list, people CRM with search/filter/drilldown to every answer, CSV exports, church onboarding with branding and per-assessment withhold, bundles, seat tiers, coupons, embeds, drag-reorder, platform fee.

### Housekeeping first

Five orphaned admin pages (`/admin/pricing`, `/admin/gifts`, `/admin/pages`, `/admin/homepage`, `/admin/embed`) duplicate the hub's features, are not in the nav, but are still live and reachable by URL. They will drift out of sync, and in one case already have: the **fuller** coupon form (applies-to, max uses, expiry) lives on the hidden page while the nav'd hub has the reduced one. Delete or redirect them and move the full coupon form into the hub. **WO-13.**

### Missing admin power, ranked by value

1. **Resend report email / recover link** (small). One button on the person drilldown.
2. **Real revenue reporting** (medium). "Est. revenue" multiplies grants by current list price, ignoring tiers, coupons, and fees. Store `amount_cents` on the grant at purchase and report actuals.
3. **Refunds** (medium). One button: refund the PaymentIntent and revoke/decrement the grant. Today a Stripe-dashboard refund leaves the access code alive.
4. **Honest funnel** (medium). "Started" is currently fiction because sessions are only created at submit. Recording a start event also unlocks resume (WO-8) and real abandonment data.
5. **Audit log** (medium). Admins can read individual answers to sensitive instruments with no access trail. For pastoral data this is close to an ethical requirement.
6. **Care-list workflow** (small). "Contacted" checkbox and notes on the wellbeing list. The most mission-aligned small feature available.
7. **Roles** (medium). `is_admin` is one boolean. Wellbeing data should be care-team-only; content editing and finance should be separable.
8. **Church aggregate reporting** (medium). Gifts distribution across a congregation, participation rate, trends. This is your partnership retention feature.
9. **CSV import / bulk seat issue** (medium). Membership-class roster in, invite links out.
10. **Email log** (small). Which emails sent, which failed. Pairs with WO-9's failure logging.
11. **View-as-church impersonation** (small). Stop debugging via screenshots.

#### WO-2: Lock down results, and enforce withhold server-side

```text
In the musa-assessments repo, fix a critical vulnerability: the live RLS policies
"anon sessions readable" (sessions) and "anon results readable for tokenized sessions"
(results) let any anon-key holder enumerate every anonymous taker's result_token, scores,
and embedded PII. Also, the church "withhold results" feature is only enforced in client
React.

1. Create supabase/migration_33_tokenized_result_rpc.sql that:
   - DROPs POLICY "anon sessions readable" ON public.sessions.
   - DROPs POLICY "anon results readable for tokenized sessions" ON public.results.
   - Creates result_by_token(p_token text) RETURNS jsonb, LANGUAGE sql, SECURITY DEFINER,
     SET search_path = public. It joins sessions to results on the exact result_token match
     and returns one row with: assessment slug/name/scoring_type, scored_json, created_at,
     session church_id, and a computed boolean "withheld".
```
